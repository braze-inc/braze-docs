# 자주 묻는 질문

> 이 문서는 캠페인에 대해 자주 묻는 질문과 답변을 제공합니다.

### 멀티채널 캠페인은 어떻게 만들 수 있나요?

멀티채널 캠페인을 만들려면 **메시징** > **캠페인**을 선택합니다. 그런 다음 **캠페인 생성** > **멀티채널**을 선택합니다. 여기에서 다음 메시징 채널 중에서 선택할 수 있습니다: 콘텐츠 카드, 이메일, LINE, 푸시 알림, SMS/MMS/RCS, 웹훅 또는 WhatsApp.

### 내 멀티채널 캠페인에 대조군을 추가할 수 있나요?

아니요, 캠페인의 대조군은 이메일 A 대 이메일 B와 같은 단일 채널 메시징을 위해 설계되었습니다. 대안으로, 다른 채널, 메시징 콘텐츠 및 전달 타이밍을 테스트하기 위해 [Canvas]({{site.baseurl}}/user_guide/engagement_tools/canvas)를 사용해 보세요. 

### 캠페인을 테스트하고 최적화하려면 어떻게 시작하면 되나요?

다변량 캠페인과 여러 배리언트가 있는 캔버스를 실행하는 것이 좋은 출발점입니다! 예를 들어, [다변량 캠페인]({{site.baseurl}}/user_guide/engagement_tools/testing/multivariant_testing/)을 실행하여 문구나 제목란이 다른 하나의 메시지를 테스트할 수 있습니다. 여러 배리언트가 있는 캔버스는 전체 워크플로를 테스트하는 데 도움이 됩니다.

### 왜 내 캠페인의 열람률이 감소했나요?

낮은 열람률이 항상 기술적 문제와 관련이 있는 것은 아닙니다. 이메일 클리핑으로 인해 추적 픽셀이 누락될 수 있습니다. 그러나 콘텐츠나 오디언스 크기의 변화로 인해 이메일을 열어보는 사용자가 줄어들었을 가능성도 있습니다. 

### 캠페인 오디언스는 어떻게 평가되나요?

기본적으로 캠페인은 입장 시점에 오디언스 필터를 확인합니다. 지연이 있는 실행 기반 캠페인의 경우, 메시지가 전송될 때 사용자가 여전히 타겟 오디언스에 속해 있는지 확인하기 위해 전송 시점에 세그먼트 기준을 재평가하는 옵션이 있습니다. 

### 특정 캠페인 또는 캔버스에서 고유 수신자 수와 전송 횟수에 차이가 있는 이유는 무엇인가요?

한 가지 가능한 설명은 캠페인 또는 캔버스에 재자격이 켜져 있어, 세그먼트 및 전달 설정에 해당하는 사용자가 메시지를 두 번 이상 받을 수 있다는 것입니다. 재자격이 켜져 있지 않은 경우, 전송 횟수와 고유 수신자 간의 차이는 사용자가 여러 플랫폼에 걸쳐 프로필에 연결된 여러 기기를 가지고 있기 때문일 수 있습니다. 

예를 들어, iOS와 웹 푸시 알림이 모두 있는 캔버스가 있는 경우, 모바일과 데스크톱 기기를 모두 사용하는 특정 사용자가 두 개 이상의 메시지를 받을 수 있습니다.

### 왜 멀티채널 캠페인에서 전환 수가 고유 사용자 수를 초과할 수 있나요?

멀티채널 캠페인에서 Braze는 사용자별이 아니라 채널별로 전환을 계산합니다. 사용자가 전환 기간 내에 단일 전환 동작을 수행하면, Braze는 해당 전환을 사용자가 메시지를 받은 각 채널에 귀속시킵니다. 즉, 사용자가 여러 채널(예: 이메일과 푸시 모두)에서 메시지를 받고 전환하면, Braze는 각 채널에 대해 하나씩 여러 전환을 계산합니다. 그 결과, 총 전환 수는 전환한 고유 사용자 수를 초과할 수 있습니다.

예를 들어, 멀티채널 캠페인이 사용자에게 이메일과 푸시 알림을 모두 보내고, 그 사용자가 두 메시지를 모두 받은 후 전환 기간 내에 하나의 전환 동작을 수행하면, Braze는 이를 두 개의 전환으로 계산합니다. 하나는 이메일에 귀속되고 다른 하나는 푸시에 귀속됩니다. 같은 사용자의 단일 동작임에도 불구하고 말입니다.

### 왜 내 캠페인의 도달 가능한 사용자 기반이 캠페인에 사용하는 세그먼트보다 더 작은가요?

[글로벌 컨트롤 그룹]({{site.baseurl}}/user_guide/engagement_tools/testing/global_control_group/)이 설정되어 있는 경우, 도달 가능한 오디언스의 일정 비율이 캠페인을 받지 못하게 됩니다. 이는 캠페인이 동일한 세그먼트를 사용하더라도 세그먼트의 도달 가능한 사용자 수가 캠페인의 도달 가능한 사용자 수보다 많을 수 있음을 의미합니다.

### 현지 시간대 전달은 무엇을 제공하나요?

현지 시간대 전달을 사용하면 사용자의 개별 시간대를 기준으로 메시징 캠페인을 세그먼트에 전달할 수 있습니다. 현지 시간대 전달을 사용하지 않으면 캠페인은 Braze에서 회사의 시간대 설정에 따라 스케줄됩니다. 

예를 들어, 런던에 본사를 둔 회사가 오후 12시에 캠페인을 보내면 미국 서부 해안의 사용자에게는 새벽 4시에 도달하게 됩니다. 앱이 특정 국가에서만 사용 가능한 경우 이것이 문제가 되지 않을 수 있습니다. 그렇지 않으면, 사용자 기반에 이른 아침 푸시 알림을 보내는 것을 피하는 것을 강력히 권장합니다.

### Braze는 사용자의 시간대를 어떻게 인식하나요?

Braze는 사용자의 기기에서 자동으로 시간대를 결정합니다. 이를 통해 시간대 정확성과 사용자의 완전한 커버리지를 보장합니다. 사용자 API를 통해 생성되었거나 시간대가 없는 사용자는 SDK에서 앱을 통해 인식될 때까지 회사의 시간대가 기본 시간대로 사용됩니다. 

대시보드의 [회사 설정]({{site.baseurl}}/user_guide/administrative/app_settings/company_settings/)에서 회사의 시간대를 확인할 수 있습니다.

### Braze는 현지 시간대 전달을 위해 언제 사용자를 평가하나요?

Braze는 다음 시점에서 사용자의 입장 자격을 평가합니다:

- 사모아 시간(UTC+13) 또는 일광 절약 시간 동안 UTC+14
- 스케줄된 날의 현지 시간

사용자가 입장 자격을 얻으려면 두 가지 검사를 모두 통과해야 합니다. 예를 들어, 캔버스가 2021년 8월 7일 오후 2시(현지 시간 기준)에 시작될 예정인 경우, 뉴욕에 위치한 사용자를 타겟팅하려면 다음과 같은 자격 검사가 필요합니다:

- 2021년 8월 6일 오후 9시 뉴욕
- 2021년 8월 7일 오후 2시 뉴욕

사용자는 시작 24시간 전에 세그먼트에 속해 있어야 합니다. 사용자가 첫 번째 검사에서 자격이 없으면 Braze는 두 번째 검사를 시도하지 않습니다.

예를 들어, 캠페인이 오후 7시 UTC에 전달되도록 스케줄된 경우, 시간대(예: 사모아)가 식별되면 캠페인 전송을 즉시 대기줄에 추가하기 시작합니다. 이는 캠페인을 보내는 것이 아니라 메시지를 보낼 준비를 하고 있다는 의미입니다. 자격을 확인할 때 사용자가 어떤 필터에도 일치하지 않으면 타겟 오디언스에 포함되지 않습니다.

또 다른 예로, 같은 날 발송될 두 개의 캠페인을 만들고 싶다고 가정해 보겠습니다. 하나는 아침에, 하나는 저녁에 발송되며, 사용자가 첫 번째 캠페인을 이미 받은 경우에만 두 번째 캠페인을 받을 수 있도록 필터를 추가합니다. 현지 시간대 전달을 사용하면 일부 사용자가 두 번째 캠페인을 받지 못할 수 있습니다. 이는 사용자의 시간대가 확인될 때 자격을 검사하기 때문에, 해당 시간대에서 아직 스케줄된 시간이 되지 않았다면 첫 번째 캠페인을 받지 못한 것이므로 두 번째 캠페인에 자격이 없게 됩니다.

### 현지 시간대 캠페인은 어떻게 스케줄하나요?

캠페인을 스케줄할 때 지정된 시간에 보내도록 선택한 다음 **현지 시간대의 사용자에게 캠페인 보내기**를 선택합니다.

Braze는 모든 현지 시간대 캠페인을 24시간 전에 스케줄하는 것을 강력히 권장합니다. 이러한 캠페인은 하루 전체에 걸쳐 전송되어야 하므로, 24시간 전에 스케줄하면 메시지가 전체 세그먼트에 도달할 수 있습니다. 그러나 필요한 경우 24시간 미만 전에도 스케줄할 수 있습니다. Braze는 전송 시간을 1시간 이상 놓친 사용자에게는 메시지를 보내지 않는다는 점을 유의하세요. 

예를 들어, 현재 오후 1시이고 현지 시간대 캠페인을 오후 3시로 스케줄하면, 캠페인은 현지 시간이 오후 3시에서 오후 4시 사이인 모든 사용자에게 즉시 전송되지만, 현지 시간이 오후 5시인 사용자에게는 전송되지 않습니다. 또한 캠페인에 대해 선택한 전송 시간은 회사 시간대에서 아직 지나지 않은 시간이어야 합니다.

24시간 미만 전에 스케줄된 현지 시간대 캠페인을 편집해도 메시지의 스케줄은 변경되지 않습니다. 현지 시간대 캠페인을 편집하여 나중에 전송하기로 결정한 경우(예: 오후 6시 대신 오후 7시), 원래 전송 시간을 선택할 때 타겟 세그먼트에 있던 사용자는 여전히 원래 시간(오후 6시)에 메시지를 받게 됩니다. 현지 시간대를 편집하여 더 이른 시간(예: 오후 5시 대신 오후 4시)에 전송하는 경우, 캠페인은 여전히 원래 시간(오후 5시)에 모든 세그먼트 멤버에게 전송됩니다. 

{% alert note %}
캔버스 구성요소의 경우, 현지 시간대 전달을 위해 사용자 여정의 다음 구성요소를 받기 위해 해당 구성요소에 24시간 동안 머물러 있을 필요가 없습니다. 
{% endalert %}

사용자가 캠페인에 다시 참여할 수 있도록 허용한 경우, 원래 시간(오후 5시)에 다시 캠페인을 받게 됩니다. 그러나 이후의 모든 캠페인 발송에 대해서는 업데이트된 시간에만 메시지가 전송됩니다.

### 현지 시간대 캠페인의 변경 사항은 언제 적용되나요?

현지 시간대 캠페인의 타겟 세그먼트에는 전체 세그먼트에 전달을 보장하기 위해 시간 기반 필터에 최소 48시간의 기간이 포함되어야 합니다. 예를 들어, 다음 필터를 사용하여 둘째 날 사용자를 타겟팅하는 세그먼트를 생각해 보겠습니다:

- 앱을 처음 사용한 지 1일 이상 경과
- 앱을 처음 사용한 지 2일 미만

현지 시간대 전달은 전달 시간과 사용자의 현지 시간대에 따라 이 세그먼트의 사용자를 놓칠 수 있습니다. 이는 사용자의 시간대에서 전달이 트리거되는 시점까지 사용자가 세그먼트를 벗어날 수 있기 때문입니다.

### 시작 전에 스케줄된 캠페인에 어떤 변경을 할 수 있나요?

캠페인이 스케줄되면, 메시지를 전송하기 위해 대기줄에 추가하기 전에 메시지 구성을 제외한 모든 항목을 편집해야 합니다. 모든 캠페인과 마찬가지로, 시작 후에는 전환 이벤트를 편집할 수 없습니다.

### 스케줄된 캠페인을 업데이트했는데 왜 시작되지 않았나요?

이것은 캠페인이 업데이트된 정확한 시간에 시작되도록 스케줄된 경우 발생할 수 있습니다. 예를 들어, 현재 오후 3시 10분인데 캠페인을 오후 3시 10분에 시작하도록 변경하고 **캠페인 업데이트**를 선택했다면, 이미 오후 3시 10분이 지났으므로 시작 스케줄 시간이 지난 것입니다. 동일한 시간에 캠페인을 스케줄하는 대신, **캠페인 시작 시 즉시 발송**을 선택하세요.

### 스케줄된 캠페인에서 메시지가 대기줄에 추가되기 전의 "안전 구역"은 무엇인가요?

다음 시간 내에 메시지를 변경하는 것을 권장합니다:

- **일회성 스케줄 캠페인:** 스케줄된 전송 시간까지 편집 가능합니다.
- **반복 스케줄 캠페인:** 스케줄된 전송 시간까지 편집 가능합니다.
- **현지 전송 시간 캠페인:** 스케줄된 전송 시간 24시간 전까지 편집 가능합니다.
- **최적 전송 시간 캠페인:** 캠페인이 전송되도록 스케줄된 날 24시간 전까지 편집 가능합니다.

이러한 권장 사항 외의 시간에 메시지를 변경하면, 전송된 메시지에 업데이트가 반영되지 않을 수 있습니다. 예를 들어, 캠페인이 현지 시간 오후 12시에 전송되도록 스케줄된 상태에서 3시간 전에 전송 시간을 편집하면 다음과 같은 상황이 발생할 수 있습니다:

- Braze는 전송 시간을 1시간 이상 놓친 사용자에게 메시지를 보내지 않습니다.
- 이미 대기줄에 추가된 메시지는 조정된 시간이 아닌 원래 대기줄에 추가된 시간에 전송될 수 있습니다.

변경이 필요하면 현재 캠페인을 중지하는 것이 좋습니다(이렇게 하면 대기줄에 있는 메시지가 취소됩니다). 그런 다음 캠페인을 복제하고 필요에 따라 변경한 후 새 캠페인을 시작할 수 있습니다. 이미 첫 번째 캠페인을 받은 사용자를 이 캠페인에서 제외해야 할 수도 있습니다. 시간대 전송이 가능하도록 캠페인 스케줄 시간을 다시 조정해야 합니다.

### 일광 절약 시간(DST) 전환일에 왜 아무 사용자도 내 일일 스케줄 캠페인에 참여하지 않았나요?

일광 절약 시간(DST) 전환일에는 일일 스케줄 캠페인이 평소보다 최대 1시간 더 일찍 또는 늦게 실행될 수 있으며, 시계가 앞으로 가는지 뒤로 가는지에 따라 달라집니다. 세그먼트가 스케줄된 전송 시간의 1시간 이내에 타임스탬프가 있는 커스텀 속성이나 이벤트에 의존하는 경우, 캠페인이 DST 날에 자격을 평가할 때 해당 사용자가 아직 자격을 갖추지 못할 수 있습니다.

예를 들어, 사용자가 일반적으로 UTC 오후 3시에 커스텀 속성 업데이트를 받고, 캠페인이 뉴욕(동부 시간)에서 매일 오전 10시 30분에 실행된다고 가정해 보겠습니다. 뉴욕이 표준 시간(UTC-5)일 때, 오전 10:30 ET는 UTC 오후 3:30에 해당하므로 캠페인은 속성이 기록된 후에 실행됩니다. 뉴욕이 일광 절약 시간(UTC-4)으로 전환되면, 오전 10:30 ET는 UTC 오후 2:30에 해당하므로, 봄으로 전환되는 DST 날에는 캠페인이 UTC 오후 3시 속성 업데이트 전에 실행될 수 있습니다. 자격을 부여하는 속성이 아직 존재하지 않기 때문에 해당 사용자들은 필터링됩니다. 재자격이 꺼져 있으면, 이전 날에 입장한 사용자들은 다시 입장할 수 없으며, 그 날의 입장 수는 0이 됩니다.

이를 피하려면, 커스텀 속성이나 이벤트 업데이트가 캠페인의 스케줄된 전송 시간보다 1시간 이상 전에 발생하도록 하세요.

### 캠페인에 입장하는 사용자 수가 예상 수와 일치하지 않는 이유는 무엇인가요?

캠페인에 입장하는 사용자 수는 오디언스와 트리거가 평가되는 방식에 따라 예상 수와 다를 수 있습니다. Braze에서는 트리거 전에 오디언스를 평가합니다([속성 변경]({{site.baseurl}}/user_guide/engagement_tools/campaigns/building_campaigns/delivery_types/triggered_delivery/attribute_triggers/#change-custom-attribute-value) 트리거를 사용하지 않는 한). 이로 인해 트리거 동작이 평가되기 전에 사용자가 선택한 오디언스에 속하지 않으면 캠페인에서 이탈하게 됩니다.

{% alert tip %}
캠페인 문제 해결에 대한 추가 지원이 필요한 경우, 최근 30일간의 진단 로그만 보유하고 있으므로 문제 발생일로부터 30일 이내에 Braze 고객지원팀에 문의하시기 바랍니다.
{% endalert %}

### 내 캠페인 분석 페이지에서 CSV 내보내기 사용자 데이터와 CSV 내보내기 이메일 주소 옵션의 차이는 무엇인가요?

**CSV 내보내기 이메일 주소** 옵션을 선택하면 이메일 주소가 있는 사용자의 데이터만 다운로드됩니다. 예를 들어, 100,000명의 사용자 세그먼트가 있지만 그 중 50,000명만 이메일 주소를 가지고 있고 **CSV 내보내기 이메일 주소**를 클릭하면, 내보내기에는 50,000개의 데이터 행만 포함됩니다. 반면, **CSV 내보내기 사용자 데이터**를 선택하면 모든 사용자 데이터가 내보내집니다.

### 캠페인의 API 식별자로 검색할 수 있나요?

네, **캠페인** 페이지에서 `api_id:YOUR_API_ID` 필터를 사용하여 API 식별자로 캠페인을 검색할 수 있습니다. 자세한 내용은 [캠페인 검색]({{site.baseurl}}/user_guide/engagement_tools/campaigns/managing_campaigns/search_campaigns/)을 참조하세요.

### 입력 필드와 표시된 텍스트에서 공백이 다르게 나타나는 이유는 무엇인가요? 

공백 처리는 CSS 스타일링 때문에 입력 필드와 표시된 텍스트 구성요소 간에 다릅니다. 기본 `white-space: normal` CSS가 적용된 텍스트 구성요소에서는 여러 개의 연속된 공백이 표시될 때 하나의 공백으로 축소됩니다. 이는 렌더링된 텍스트에 대한 표준 HTML 동작입니다. 

입력 필드는 정확한 데이터 입력을 위해 입력한 대로 여러 개의 공백을 그대로 유지합니다. 이는 여러 개의 공백이 있는 텍스트가 입력 필드(모든 공백이 유지됨)에서 볼 때와 대시보드의 다른 부분(CSS가 여러 개의 공백을 축소할 수 있음)에서 표시될 때 다르게 나타날 수 있음을 의미합니다. 

예를 들어, 입력 필드에 여러 개의 공백이 있는 캠페인 이름이나 UTM 매개변수를 입력하면 모든 공백이 유지됩니다. 그러나 동일한 텍스트가 검색 결과, 캠페인 목록 또는 다른 텍스트 구성요소에 나타날 때, CSS 공백 처리로 인해 여러 개의 공백이 하나의 공백으로 나타날 수 있습니다. 

### API 캠페인과 API 트리거 캠페인의 차이점은 무엇인가요?

API 트리거 캠페인을 사용하면 자체 서버 및 시스템에서 콘텐츠 전달을 트리거하면서 Braze 대시보드 내에서 캠페인 문구, 다변량 테스트 및 재자격 규칙을 관리할 수 있습니다. 이러한 메시지에는 실시간으로 메시지에 템플릿을 적용할 추가 데이터를 포함할 수도 있습니다.

API 캠페인은 API를 사용하여 전송된 메시지를 추적하는 데 사용됩니다. 대부분의 캠페인과 달리 메시지, 수신자 또는 스케줄을 지정하지 않고 대신 식별자를 API 호출에 전달합니다.

### "요청 시간 초과" 오류에 대한 고객지원 티켓을 제출할 때 무엇을 포함해야 하나요?

캠페인 또는 캔버스를 생성하거나 편집하는 중에 "요청 시간 초과" 오류가 발생하여 [Braze 고객지원]({{site.baseurl}}/braze_support/)에 문의해야 하는 경우, 해결 속도를 높이기 위해 다음 정보를 포함하세요:

- **화면 녹화:** 오류가 나타나기 전에 수행한 단계를 녹화한 것으로, 페이지 전환을 포함합니다.
- **타임스탬프 및 시간대:** 오류가 발생한 정확한 시간과 시간대입니다.
- **브라우저 및 버전:** 사용 중인 브라우저(예: Chrome 120, Safari 17)와 다른 브라우저에서 오류를 재현해 보았는지 여부입니다.
- **재현 단계:** 오류를 트리거하는 동작에 대한 명확한 설명으로, 관련된 특정 캠페인 또는 캔버스 설정을 포함합니다.
- **네트워크 로그(선택 사항):** 브라우저 개발자 도구(**네트워크** 탭)를 열고 오류를 재현한 다음 네트워크 로그를 HAR(HTTP Archive) 로그로 내보냅니다. 이를 통해 고객지원팀이 어떤 API 호출이 시간 초과되는지 식별하는 데 도움이 됩니다.

### 전송 분석이 설정한 최대 수신자 제한과 일치하지 않는 이유는 무엇인가요?

활성 캠페인에 최대 수신자 제한을 추가하거나 변경하면, 다음과 같은 이유로 전송 분석에 제한이 반영되지 않을 수 있습니다:

- **시작 후 제한 추가:** 캠페인 시작 시 최대 수신자 제한이 설정되지 않은 경우, 제한을 적용하기 전에 이미 대기줄에 추가된 메시지는 여전히 전송됩니다. 제한은 변경 사항을 저장한 후 대기줄에 추가하는 전송에만 적용됩니다.
- **사용량 제한과의 상호작용:** 캠페인에 사용량 제한도 적용된 경우, 메시지가 더 긴 시간에 걸쳐 분배될 수 있습니다. 최대 수신자 제한은 메시지가 대기줄에 추가될 때 평가되며, 전달될 때가 아닙니다. 메시지가 이미 대기줄에 있는 동안 제한이 변경되면, 해당 메시지에는 원래 제한이 적용됩니다.
- **반복 캠페인:** 반복 캠페인의 경우, 각 스케줄된 전송은 최대 수신자 제한을 독립적으로 평가합니다. 전송 간에 제한을 변경해도 이전 전송 횟수가 소급 조정되지 않습니다.

불일치를 방지하려면, 캠페인을 시작하기 전에 최대 수신자 제한을 설정하고 전송이 진행 중인 동안에는 수정하지 마세요.
